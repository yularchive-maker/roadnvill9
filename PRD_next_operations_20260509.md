## 2. 운영 원칙

- 이 시스템은 고객용 예약 사이트가 아니라 내부 운영관리 웹앱이다.
- Telegram은 1차 업체 가능 여부 확인 채널로만 사용한다.
- 고객 안내 문자, 전화, 오프라인 안내는 운영자가 직접 처리한다.
- 웹앱은 고객 안내 발송 여부와 최종 예약확정 상태만 기록한다.
- 추후 Kakao 전환 가능성을 고려해 회신 상태와 확정 로직은 채널에 종속되지 않게 유지한다.
- 고객 연락처, 결제금액, 내부 메모, 내부 정산 단가는 업체 메시지에 포함하지 않는다.

## 3. 다음 작업: 업체 Telegram 연결 관리

### 목표

기준정보의 체험 업체 관리 화면에서 각 업체의 Telegram 연결 상태를 바로 확인하고 수정할 수 있게 한다.

### 수정 대상

- `app/dashboard/master/page.js`

### 기능

- 업체 카드에 Telegram 연결 여부 표시
- 연결된 경우 `telegram_chat_id` 표시
- 업체 추가/수정 모달에서 `telegram_chat_id` 입력
- 업체 추가/수정 모달에서 `telegram_username` 입력
- `telegram_chat_id`가 입력되면 `telegram_linked_at` 기록
- `telegram_chat_id`가 비워지면 연결 해제 상태로 처리

### 테스트 방법

1. `/dashboard/master` 이동
2. 체험 업체 탭에서 업체 카드의 Telegram 연결 상태 확인
3. 업체 수정 모달 열기
4. `telegram_chat_id` 입력 후 저장
5. 카드에 `텔레그램 연결됨` 표시 확인
6. 해당 업체가 포함된 예약에서 Telegram 요청 발송 확인

### 예상 위험

- 업체가 봇에 `/start`를 보내지 않았거나 chat_id가 틀리면 발송 실패
- 운영자가 username과 chat_id를 혼동할 수 있으므로 화면에서는 chat_id를 우선값으로 관리

## 4. 다음 후보 작업

### 4.1 Telegram 메시지 가독성 개선

업체가 휴대폰에서 빠르게 판단할 수 있도록 Telegram 메시지는 카테고리별로 명확하게 나눈다.

예약 가능 여부 확인 요청 메시지 구성:

- `📌 예약 가능 여부 확인 요청`
- `📍 요청 정보`
- `🗓 일정/장소`
- `👥 인원 현황`
- `📋 업체 당일 일정`
- `✅ 확인 기준`
- `⏰ 회신 마감`

추후 업체 최종 안내 메시지를 만들 경우 구성:

- `📌 예약 최종 확정 안내`
- `📍 확정 정보`
- `🗓 확정 일정/장소`
- `👥 확정 인원`
- `📋 준비/운영 참고사항`
- `☎️ 문의/변경 시 연락 기준`

메시지 작성 원칙:

- 고객 연락처, 결제금액, 할인금액, 내부 정산 단가, 내부 메모는 포함하지 않는다.
- 업체가 수행 판단과 준비에 필요한 정보만 보낸다.
- Telegram 단계에서는 고객 안내 자동 발송 기능을 만들지 않는다.
- 추후 Kakao 전환 시에도 같은 카테고리 구조를 유지한다.

### 4.4 체험 업체 목록 UI 밀도 전환 기준

현재는 업체 수가 많지 않으므로 업체 카드에서 대표 체험과 단가를 함께 보여준다.

현재 표시 방식:

- 업체명
- 업체 KEY
- 담당자
- 연락처
- Telegram 연결/미연결
- 대표 체험 최대 3개
- 각 체험별 단가와 정산 방식
- 체험이 더 많으면 전체보기로 업체 수정 모달에서 확인

업체 수가 많아질 경우에는 목록 화면을 고밀도 명부형으로 전환한다.

전환 기준 예시:

- 체험 업체가 20개 이상으로 늘어나는 경우
- 한 업체가 보유한 체험 프로그램이 평균 4개 이상으로 늘어나는 경우
- 기준정보 화면에서 스크롤이 길어져 업체를 빠르게 찾기 어려운 경우

고밀도 명부형 표시 방식:

- 업체명
- 업체 KEY
- 대표자/담당자
- 연락처
- Telegram 연결/미연결
- 등록 체험 있음/없음
- 상세보기 버튼

고밀도 명부형에서는 체험명과 단가를 목록 카드에 직접 나열하지 않는다.
전체 체험 목록, 체험별 단가, 정산 방식, 체험 추가/수정/삭제는 업체 상세보기/수정 모달에서 관리한다.

목표:

- 업체 목록은 빠르게 찾고 비교하는 용도로 유지
- 체험과 단가는 상세 관리 영역에서 정확하게 확인
- 업체 수가 늘어나도 기준정보 화면이 복잡해지지 않게 유지

### 4.5 재발송 기준 고도화

이미 회신된 업체는 기본적으로 재발송하지 않는다.

재발송이 필요한 경우:

- 요청 시간 변경
- 장소/구역 변경
- 프로그램 변경
- 인원 변경
- 운영자가 직접 재요청 선택

현재 방향:

- 가능이 아닌 건은 운영자가 업체와 조율 후 수동 회신 입력
- 조율 결과 확정되면 수동으로 가능 처리
- 이미 회신된 업체는 일반 Telegram 요청에서 자동 제외
- 시간, 장소, 인원 등 정보 변경 후 다시 확인해야 할 때는 예약 상세의 `선택 재요청` 버튼 사용
- `선택 재요청`은 기존 회신 상태를 `회신대기`, 최종 판단을 `미회신`으로 되돌린 뒤 Telegram 메시지를 다시 발송

### 4.6 Kakao 전환 대비

Telegram 전용 로직을 최소화하고, 회신 결과는 공통 상태값으로 유지한다.

공통 상태:

- 가능
- 불가능
- 시간조정 필요
- 인원조정 필요
- 보류

추후 Kakao 단계에서 추가할 수 있는 필드:

- `kakao_template_id`
- `kakao_message_id`
- `kakao_sent_at`
- `kakao_callback_payload`

## 5. 배포 전 체크리스트

- `.env.local`은 git에 포함하지 않는다.
- `.env.example`에는 변수명만 남기고 실제 키 값은 넣지 않는다.
- Vercel 환경변수에 서버 전용 키를 등록한다.
- `SUPABASE_SERVICE_ROLE_KEY`는 서버 라우트에서만 사용한다.
- `TELEGRAM_BOT_TOKEN`은 서버 라우트에서만 사용한다.
- `TELEGRAM_WEBHOOK_SECRET`을 설정하고 Telegram webhook 요청의 `x-telegram-bot-api-secret-token` 헤더를 검증한다.
- 운영 배포 전 테스트로 노출된 Telegram token은 revoke 후 새 token 사용을 권장한다.
- Vercel 배포 후 Telegram webhook URL을 운영 URL 기준으로 등록한다.

### 5.1 Telegram webhook 운영 API

내부 로그인 상태에서만 호출 가능한 webhook 관리 API를 둔다.

- `GET /api/telegram/webhook-settings`: 현재 Telegram webhook 정보 확인
- `POST /api/telegram/webhook-settings`: 현재 배포 URL 기준으로 webhook 등록
- `DELETE /api/telegram/webhook-settings`: webhook 해제

운영 등록 기준 URL:

- `https://roadnvill9.vercel.app/api/telegram/webhook`

주의:

- `/api/telegram/webhook` 자체는 Telegram이 호출해야 하므로 로그인 예외 경로로 유지한다.
- 대신 `TELEGRAM_WEBHOOK_SECRET`으로 Telegram 요청인지 검증한다.
- webhook 관리 API는 로그인한 내부 직원만 호출 가능하다.

### 5.2 Telegram webhook 운영 화면

기준정보 > 체험 업체 탭에서 webhook 운영 상태를 확인하고 관리할 수 있다.

기능:

- 현재 webhook URL 확인
- 대기 업데이트 수 확인
- 마지막 webhook 오류 메시지 확인
- 현재 배포 URL 기준 webhook 등록
- webhook 해제

주의:

- 운영 중 webhook을 해제하면 업체가 버튼을 눌러도 회신이 자동 저장되지 않을 수 있다.
- Vercel 운영 환경변수에 `TELEGRAM_WEBHOOK_SECRET`이 없으면 webhook 등록은 실패해야 한다.
